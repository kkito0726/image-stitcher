# PDR: プレビュー即時返却 + フル解像度の遅延ダウンロード

- ステータス: Draft
- 作成日: 2026-07-05
- 対象: `POST /stitch` のレスポンス方式変更(`backend/src/`)、結果表示・ダウンロード動線(`frontend/src/`)
- 関連文書:
  - [PDR-python-backend-modernization.md](./PDR-python-backend-modernization.md)(実装済み。本 PDR はそのレスポンス方式に変更を加える)
  - [PDR-zero-cost-hosting.md](./PDR-zero-cost-hosting.md)(§4 の転送コスト試算を本 PDR の実測値で更新する)

## 1. 背景

0 円ホスティング PDR の検討中、実運用のメジャーユースケース(2.8MB JPEG × 6 枚)を想定して実測したところ、2 つの課題が判明した。

1. **合成結果のフル解像度 PNG が非常に大きい**: 実測 73.8MB(20699×3456、71MP)。Cloud Run 運用時、この転送だけで下り無料枠(月 1GB)を月 13〜14 回で使い切る
2. **フル PNG のエンコード自体が重い**: 実測約 1.4 秒。ローカル運用でもレスポンス時間(合計約 2.9 秒)の約半分をこのエンコードが占めている

いずれも「毎回フル解像度を生成・転送する」設計に起因する。実際にはユーザーの多くは画面プレビューを見るだけで、フル解像度を必要とするのは研究データとして保存する場合など一部に限られる。

## 2. 目的

- プレビュー表示に必要な分だけ即座に返し、フル解像度は要求(ダウンロード)時のみ生成・転送する
- **ローカル運用**: レスポンスタイムを短縮する(実測 約 2.9 秒 → 約 1.5 秒、フル PNG エンコードを遅延させることで達成。ネットワーク起因ではない)
- **クラウド運用**: 転送コストを削減し、0 円運用の実質的な余裕を大きく広げる

## 3. 実測データ(2026-07-05、6 枚 × 2.8MB JPEG 入力、Apple Silicon ローカル環境)

| 項目 | 実測値 |
|---|---|
| 入力合計(アップロード) | 17.8MB |
| 合成結果の解像度 | 20699×3456(71MP) |
| フル PNG サイズ | 73.8MB |
| JPEG decode | 267ms |
| stitch 本体 | 1215ms |
| フル PNG encode | 1408ms |
| **合計(現行方式)** | **約 2890ms** |

プレビュー候補の比較(合成結果を縮小 + JPEG 化):

| 解像度 / 品質 | サイズ | フル比 | encode 時間 |
|---|---|---|---|
| 2560px / q85 | 595KB | 1/124 | 4ms |
| **1920px / q80(採用)** | **353KB** | **1/209** | **2ms** |
| 1280px / q80 | 137KB | 1/537 | 1ms |

1920px は一般的な画面表示に十分な解像度で、転送量とのバランスが良いため既定値とする。

計測はテクスチャの多い合成画像によるもので、実際の顕微鏡画像では圧縮率が変わる可能性がある(§10)。

## 4. API 契約の変更

Base64 in JSON を避ける既定方針(近代化 PDR)を維持しつつ、プレビューと結果 ID を返す。

### 4.1 `POST /stitch`(変更)

```
200
Content-Type: image/jpeg
X-Result-Id: <uuid>
Body: <プレビューJPEGバイナリ (1920px, q80)>
```

- `X-Result-Id` を JS から読めるよう CORS に `Access-Control-Expose-Headers: X-Result-Id` を追加する
- 422(合成失敗)/ 400(バリデーション)/ 500 のレスポンス形式は変更しない

### 4.2 `GET /stitch/{resultId}/download`(新規)

```
Request:
  Header: X-Download-Password: <password>   (Settings.download_password 設定時のみ必須)

200 image/png (フル解像度バイナリ)
401 {"error": "パスワードが違います"}          (パスワード必須時のみ発生)
404 {"error": "結果が見つからないか有効期限切れです"}
```

- フル解像度は初回アクセス時に(キャッシュされた生画像から)エンコードして返す
- 複数回ダウンロードされても再エンコードで問題ない想定(低頻度のため最適化しない)
- パスワード検証はキャッシュ参照・エンコードより前に行う(不正な試行のコストを最小化するため)

### 4.3 フル解像度ダウンロードのパスワード保護(本番環境のみ)

**背景**: プレビューは誰でも閲覧できてよいが、フル解像度(研究データとして使える品質)は「身内」のみに限定し、URL を知っただけの野良ユーザーには渡したくない、という要件。

**設計**: `Settings.download_password`(環境変数 `DOWNLOAD_PASSWORD` から読み込み)が**設定されている場合のみ** `X-Download-Password` ヘッダーの一致を要求する。

- ローカル Docker 開発環境: `DOWNLOAD_PASSWORD` を設定しない(既定)→ チェックはスキップされ、開発時に余計な手間がかからない
- Cloud Run 本番環境: `DOWNLOAD_PASSWORD` を Secret Manager 経由の環境変数として設定する → 自動的にパスワード必須になる

「本番かどうか」を判定するフラグを別途持たず、**シークレットが設定されているかどうかだけ**で振る舞いが決まる、12-factor 的なミニマル構成にする。

- 比較は `secrets.compare_digest` によるタイミングセーフ比較を用いる
- パスワードはフロントエンドのビルドに埋め込まない(public リポジトリ・public サイトのため、埋め込むと誰でも閲覧ソースから読める。埋め込んだ時点で保護の意味がなくなる)
- フロントエンドはまずパスワードなしで `GET /download` を試行し、401 が返った場合のみプロンプトで入力を求めて `X-Download-Password` を付けて再試行する(§7)。これによりローカル/本番を問わず同じビルドで動作する

**この方式が守れる範囲(正直な評価)**:

- 「URL を知っただけの野良ユーザーがフル解像度を落とす」ことは防げる(目的どおり)
- ただし本格的な攻撃者(パスワード総当たり等)への耐性は、レート制限を追加実装しない限り弱い。緩和策として**十分に長いランダム文字列**(例: `openssl rand -base64 24`)を使うことを前提とし、覚えやすい単語は避ける
- パスワードを知っている「身内」がさらに他人に教えてしまうリスクは、この方式では原理的に防げない(共有シークレット方式の限界)
- 悪用が実際に観測された場合は、レート制限や Cloud Armor 等の追加を検討する([PDR-zero-cost-hosting.md](./PDR-zero-cost-hosting.md) §7 の「悪用の実観測をトリガーに追加」という既存方針と整合)

## 5. アーキテクチャ

クリーンアーキテクチャの構造を維持し、新しい port を 1 つ追加する。

```
domain/
  ports.py          # + StitchResultCache (Protocol): put(image) -> str, get(id) -> DecodedImage | None
usecase/
  stitch_images.py  # 変更: 成功時にプレビューJPEGを生成し、cache.put()でresultIdを発行
  get_stitch_result.py  # 新規: resultIdからフル画像を取得しPNGエンコードして返す
infra/
  cache/
    in_memory_result_cache.py  # StitchResultCacheの実装 (TTL付き辞書)
presentation/
  routers/stitch.py  # レスポンスをプレビューJPEG + X-Result-Idヘッダーに変更
  routers/download.py  # 新規: GET /stitch/{resultId}/download (パスワード検証を含む)
```

- 「結果を後で取得可能にする」はユースケースの責務とし、キャッシュへの登録は `StitchImagesUseCase` 内で行う(domain/usecase は依然として `cv2` に依存しない)
- infra 層に `InMemoryResultCache` を追加。Protocol 化しておくことで、将来複数インスタンス運用になった場合に Redis 等へ差し替え可能にする(現時点ではスコープ外)
- パスワード検証は認証・認可という HTTP 層の関心事であり、domain/usecase には持ち込まず `presentation/routers/download.py` に閉じる。`Settings.download_password` を読んで比較するだけの単純なガードとする

## 6. キャッシュ設計

| 項目 | 設計 |
|---|---|
| 実装 | プロセス内メモリの辞書(`dict[str, CacheEntry]`)。`CacheEntry = (image, expires_at)` |
| TTL | 10 分 |
| 有効期限切れの扱い | アクセス時に期限チェックして削除する lazy expiry(バックグラウンドスレッド不要) |
| 保持件数上限 | 実装時に確定(§9 Phase 1)。71MP 相当の生画像 1 枚 ≈ 214MB のため、上限は 3〜5 件程度を目安に、compose のメモリ制限(2G)と gunicorn ワーカー数を踏まえて決定する |
| 複数インスタンス前提 | 0 円 PDR で決定済みの `max-instances=1` により、キャッシュ不整合は発生しない。ローカル Docker も単一コンテナのため同様に安全。**将来スケールアウトする場合はこの前提が崩れるため、その時点で共有キャッシュへの切り替えが必要**(現時点ではスコープ外) |

## 7. フロントエンド変更

- `ImgSender`: `/stitch` のレスポンス(プレビュー JPEG)を `URL.createObjectURL` で表示。レスポンスヘッダーから `resultId` を読み取り state に保持
- 「ダウンロード」ボタン: 現行の直リンクから、クリック時に `GET /api/stitch/{resultId}/download` を fetch してフル解像度 PNG を取得する方式に変更(クリック時に初めてフル解像度の転送が発生する)
- TTL 切れ(404)の場合は「有効期限が切れました。もう一度合成してください」等のメッセージを表示
- **パスワード対応**: まずパスワードなしでリクエストし、401 が返った場合のみパスワード入力(簡易プロンプト)を求めて `X-Download-Password` ヘッダー付きで再試行する。入力したパスワードは `sessionStorage` に保持し、同一セッション内での再入力を省略する。ローカル開発(パスワード未設定)では 401 が返らないため、この UI は一切表示されない

## 8. コスト試算(0 円 PDR §4 の更新)

想定シナリオ(月 300 回合成、うち 20%(60 回)がフルダウンロード):

```
プレビュー: 300回 × 353KB ≈ 106MB
フルDL:      60回 × 73.8MB ≈ 4,428MB
合計:            ≈ 4.5GB
無料枠(1GB)超過分: 約3.5GB
概算コスト: 3.5GB × 約15〜18円/GB ≈ 月50〜60円
```

プレビューのみの利用(ダウンロードなし)であれば **月 2,800 回相当まで実質 0 円**。フルダウンロードを毎回行う極端なケースでも、無料枠だけで月 13〜14 回分は賄える。想定規模(月数百回)であれば、実費は 0 円〜数十円/月に収まる見込み。

## 10. リスクと留保事項

| リスク / 留保 | 対策・備考 |
|---|---|
| メモリ枯渇(複数の生画像キャッシュが同時に滞留) | 保持件数上限 + TTL 短めで対応。§9 Phase 1 で実測に基づき確定 |
| `max-instances=1` 前提が将来崩れる | その時点で共有キャッシュ(Redis 等)へ切り替え。現時点ではスコープ外と明記 |
| ダウンロード連打による再エンコードの重複 | 許容(低頻度想定のため最適化しない) |
| プレビュー解像度が研究用途の確認に不十分な可能性 | 1920px は通常十分。必要なら解像度を上げるオプションを将来検討 |
| 実測がテクスチャ合成画像に基づく | 実際の顕微鏡画像で圧縮率が異なる可能性がある。本番投入前に実データでの再測定を推奨 |
| GCP の下り無料枠(月 1GB)の金額・料率 | 0 円 PDR 既知の留保どおり、実装・デプロイ時に最新の料金表で再確認する |
| パスワード総当たり攻撃 | レート制限は実装しない。十分に長いランダムパスワードで緩和。悪用が観測されたら追加対策を検討([PDR-zero-cost-hosting.md](./PDR-zero-cost-hosting.md) §7 と同方針) |
| パスワードの又聞き・転送 | 共有シークレット方式の原理的な限界。厳密なアクセス制御が必要になった場合は個別トークン発行方式への移行を検討(現時点ではスコープ外) |

## 11. 決定事項

1. プレビュー既定値: 1920px / JPEG q80(353KB)— 2026-07-05 決定
2. フル PNG エンコードは遅延方式(ダウンロード時に生成)を採用 — 2026-07-05 決定
3. キャッシュ TTL: 10 分 — 2026-07-05 決定
4. キャッシュ保持上限件数: 実装時(§9 Phase 1)に確定
5. フル解像度ダウンロードは共有パスワードで保護し、本番(Cloud Run)のみ有効化する。ローカル開発では `DOWNLOAD_PASSWORD` 未設定によりチェックをスキップする — 2026-07-05 決定
